PostHog UX Case Study

UX Copy Case Study: PostHog

Reducing Friction in Feature Flag Setup

The Impact

+31% Setup completion rate
-22% Time to first deployment
-54% Support tickets

The Challenge

PostHog's feature flag creation flow was experiencing a 43% drop-off rate between initiating flag creation and deployment. User interviews revealed confusion around technical terminology and uncertainty about configuration choices, leading engineers to abandon the process or seek documentation mid-flow.

The Approach

I conducted:

  • 12 user interviews with full-stack developers
  • Heatmap analysis of the setup flow
  • Competitive audit of LaunchDarkly, Split, and Statsig
  • A/B testing of copy variations

Key insights:

  1. Users didn't understand the difference between 'multivariate' and 'boolean' flags
  2. The CTA 'Create Flag' felt premature when configuration was incomplete
  3. Error states used technical jargon ('Invalid targeting parameters')

The Solution: Before vs After

Page Header

Before
Create New Feature Flag Set up a new feature flag to control rollout
After
Add a Feature Flag Control who sees what in your product
Why it works: 'Add' feels less committal than 'Create'. The second line focuses on the outcome (control) rather than the action (rollout).

Flag Type Selection

Before
Flag Type ○ Boolean ○ Multivariate Boolean flags return true/false values. Multivariate flags return string values for A/B testing.
After
What do you need this flag to do? ○ Simple on/off switch (e.g., enable dark mode) ○ Test multiple variations (e.g., test 3 pricing tiers) Not sure? Start with a simple switch; you can always change it later.
Why it works: Real-world examples replace jargon. The reassurance line reduces anxiety about making the 'wrong' choice.

Targeting Rules

Before
Targeting Rules Add conditions to control flag visibility [+ Add Condition] Property Operator Value [______] [______] [______]
After
Who should see this? (optional) Show this flag to specific users, or skip this to show everyone. [+ Add your first rule] For example: Email contains '@yourcompany.com'
Why it works: '(optional)' removes pressure. The example provides a starting point. Empty states guide rather than intimidate.

Primary CTA

Before
[Create Flag]
After
[Save and Continue]
Why it works: 'Save' confirms progress is preserved. 'Continue' signals there's a next step (implementation instructions), reducing fear of missing configuration.

Error State

Before
⚠️ Invalid targeting parameters Check your property conditions and try again.
After
⚠️ We couldn't save this rule The property 'user_id' doesn't exist yet. Add it to your event tracking first, or remove this rule. [View Properties] [Remove Rule]
Why it works: Explains the specific problem. Offers two clear paths forward. Links directly to resolution.

The Results

After rolling out the new copy across PostHog's feature flag flow:

+31% Setup completion rate increased
(57% to 88%)
-22% Time to first deployment decreased
(8.3 min to 6.5 min)
-54% Support tickets about flag configuration dropped
+18% A/B test adoption increased
(multivariate flag usage)

Key Takeaways

  1. Jargon is expensive. Every technical term you don't explain costs you conversions.
  2. Examples > definitions. 'Boolean' means nothing; 'on/off switch' creates instant mental models.
  3. Signal progress. 'Save and Continue' outperformed 'Create', 'Submit', and 'Finish' in A/B tests.
  4. Empty states are onboarding moments. Use them to teach, not just to prompt action.

Role: UX Writer
Timeframe: 3 weeks (research, implementation, testing)
Collaborators: Product Designer, 2 Engineers, Product Manager

Linear Jellyfish Integration

Linear
Product Integration

From Tickets To Traction: Jellyfish x Linear In Real Life

By Laolu Awogbade

If you ask a VP of Engineering how things are going, the honest answer is usually some version of "busy". There are boards full of issues in Linear, slide decks from quarterly planning, spreadsheets from finance. What's hard is seeing how all of that busyness adds up to progress on the bets the company said it cared about.

The Jellyfish integration is meant to close that gap. Linear stays the place where work actually moves: issues change state, cycles start and end, projects go from pitch to shipped. Jellyfish reaches in, pulls that activity into its own view, and frames it as investment and capacity. Instead of exporting CSVs and hand-building charts, leaders can see where time and money really went, straight from the work teams were already doing.

In practical terms, very little changes day to day. Someone on the team still creates an issue in Linear, attaches it to a project, adds a label or two, and moves it through states as they work. Engineers link pull requests, product managers shuffle scope between cycles, bugs get triaged. The difference is that those small actions now have a second life. They show up in Jellyfish as part of a story about how much effort went into "stability work" versus "new product", or how much time a given team actually spent on a flagship initiative.

That matters because so many conversations at leadership level are based on memory and vibes. "We've been neglecting the core product" or "We spent half the quarter fighting fires" might feel true, but they're hard to prove.

With the integration in place, a VP can open Jellyfish and see that, say, 28% of engineering time in the last three cycles went into a reliability-focused set of Linear projects, while only 7% went into a much-hyped experiment. It becomes easier to talk in specifics instead of gut feelings.

It also changes the tone between teams. When finance or the CEO wants to know where engineering time is going, the answer doesn't have to be another bespoke report. They can look at the same Jellyfish view, grounded in the same Linear data everyone else is using. The "translation tax" between people who live in the tracker and people who live in planning meetings shrinks. A staff engineer can talk about issues and scopes. A CFO can talk about investment themes. They're pointing at the same picture.

* * *

For the people doing the work, the best part is that they don't have to learn another tool just to keep leadership happy. Linear remains the place they open first thing in the morning. They still move issues, start cycles and close projects in the way that already fits their flow. Jellyfish sits slightly upstream, quietly aggregating that activity into a frame that makes sense higher up the stack.

The quiet win is trust. When leaders can see the shape of the work in context without constantly asking for "quick" updates, they tend to interrupt less. Teams get more uninterrupted time to build. Over time, the integration makes Linear feel less like a bureaucratic chore and more like a shared cockpit: the same moves that keep the board honest also keep the whole company better aligned.

If your own leadership team had this kind of view tomorrow, what's the first uncomfortable question you'd want it to help answer about how engineering time is really being spent?

Harvey Legal AI

Legal Tech Analysis

Context, Not Just Completion: Legora, Harvey And The New Legal Stack

Legal AI tools that care about workflow and precedent rather than generic summarisation

The latest generation of legal AI isn't trying to replace lawyers with chatbots. Instead, tools like Legora and Harvey are building systems that understand legal workflows, precedent, and the actual structure of how law firms operate. This is the difference between a glorified autocomplete and software that knows what questions to ask.

Walk into any mid-sized law firm and you'll find the same technology stack: a document management system that nobody quite trusts, email threads that serve as informal case tracking, Word documents with 47 rounds of tracked changes, and a billing system that talks to nothing else. Lawyers spend more time wrangling files than analysing them.

Generic AI tools promise to "summarise documents" or "draft contracts," but they don't understand why a particular clause appears in position 12.3 instead of 8.7, or why opposing counsel's latest draft quietly shifted liability in paragraph 14. They treat legal text as just text, not as the product of hundreds of negotiation rounds, regulatory constraints, and case law evolution.

What Makes Legal AI Actually Legal

Generic LLMs Applied to Law

  • Treats all documents as equivalent text
  • No understanding of precedent or jurisdiction
  • Can't track changes across negotiation rounds
  • Doesn't know what "market standard" means
  • No concept of risk allocation or liability
  • Outputs sound plausible but lack legal rigour

Purpose-Built Legal AI (Legora, Harvey)

  • Understands document hierarchies and relationships
  • Tracks precedent citations and jurisdictional variations
  • Compares versions and highlights material changes
  • Knows firm playbooks and client preferences
  • Maps risk allocation across clause interactions
  • Integrates with existing DMS and matter management

The difference shows up in everyday use. Ask a generic model to review a credit agreement and it might spot typos or flag vague language. Ask Legora or Harvey to review the same document and they'll compare it against similar deals you've closed, identify where this lender is pushing harder than usual on financial covenants, and surface the internal memo from three years ago where your firm negotiated exactly this provision for a different client.

How They Actually Work

Document Intelligence

Not just reading PDFs, but understanding that a merger agreement references disclosure schedules that reference data rooms that contain the actual substance

Precedent Mapping

Building a graph of how similar clauses have evolved across your firm's deals, tracking which formulations survived negotiation

Workflow Integration

Living inside the tools lawyers already use rather than demanding they switch to a new platform

Team Memory

Capturing institutional knowledge that usually lives in senior associates' heads and disappears when they leave

Harvey, which launched publicly in 2023 and quickly raised significant funding, positions itself as an AI copilot embedded in lawyers' existing workflows. The interface sits alongside Word, email, and the firm's DMS. When you're drafting a motion, Harvey can pull relevant case law, check how similar arguments have performed in front of this particular judge, and suggest language that matches your firm's style.

Legora takes a slightly different approach, focusing more on contract lifecycle and matter management. The system learns from every negotiation: which changes get accepted, which trigger pushback, which clients care about which issues. Over time, it builds a playbook that's specific to your practice. A junior associate reviewing an NDA doesn't just get generic advice; they get suggestions based on how partners at their firm typically handle confidentiality clauses for this industry.

The shift isn't from "no AI" to "full automation." It's from lawyers spending 60% of their time on document archaeology to having that institutional knowledge surfaced automatically.

The Technical Stack They're Building On

What Makes This Different From ChatGPT

  • Fine-tuned models: Not generic transformers but models trained on legal corpora with specialised tokenisation for citations and clause structure
  • Knowledge graphs: Representing relationships between cases, statutes, clauses, and negotiation positions rather than just text similarity
  • Retrieval-augmented generation: Pulling from firm-specific document stores and precedent databases, not just pre-training data
  • Version control awareness: Understanding tracked changes, redlines, and negotiation history as first-class data
  • Audit trails: Every suggestion includes source attribution and confidence scoring because lawyers need to verify everything

None of this replaces judgement. When Harvey suggests that adding a cure period to a default clause is common in this jurisdiction, a lawyer still needs to decide if it's appropriate for this client's risk tolerance. When Legora flags that a buyer is asking for broader indemnification than your firm typically accepts, someone needs to assess whether this deal justifies the exception.

The value is in removing the archaeology. Instead of spending three hours finding that old merger agreement that had a similar earn-out structure, you get it surfaced in 30 seconds. Instead of trying to remember which associate worked on the last deal with this opposing firm, the system tells you and surfaces their notes. Instead of wondering if this liability cap is aggressive, you see where it sits relative to your last 50 transactions.

What This Means For Law Firms

The firms deploying these tools aren't doing it to reduce headcount. They're doing it because clients increasingly expect faster turnarounds without sacrificing quality. A partner can review twice as many deals in the same time if they're not recreating analysis that someone else did last year. Junior associates get productive faster because they're learning from actual firm precedent rather than generic training materials.

There's also a competitive angle. If your firm can answer "How have we historically handled currency risk in cross-border M&A?" in real-time during a client call, while your competitor needs to get back to them tomorrow, that's an edge.

But the technology also creates new pressures. If Harvey or Legora surfaces that your firm has negotiated this exact clause 15 different ways across 15 deals, clients start asking why there's no consistency. If the AI can flag every time someone deviates from best practice, it becomes harder to justify those deviations. The tools make previously invisible patterns very visible.

The Real Question

Legal AI that actually understands law is less about automation and more about amplification. It makes experienced lawyers more efficient and helps newer lawyers ramp up faster. The firms winning aren't the ones replacing people with software. They're the ones figuring out how to combine institutional knowledge, human judgement, and AI tools into something better than any of those pieces alone.

If you're in legal tech or thinking about it, the interesting space isn't building another contract generator. It's building systems that understand how legal work actually happens — all the informal knowledge, all the edge cases, all the "we handled this before but I can't remember where" moments. That's what separates real legal AI from autocomplete.

From a lawyer's point of view, that's a subtle but important difference from generic AI writing tools.

A standard language model can help you turn bullet points into formal paragraphs, but it doesn't really know what it is saying. Solve, by contrast, is optimised to care about things IP lawyers care about: novelty, inventive step, claim dependencies, the dance between independent and dependent claims. The product lives in the browser, so it fits into the unglamorous reality of patent work. How Solve Intelligence Differs:

Traditional AI tools treat patents as text documents. Solve Intelligence treats them as interconnected graphs of concepts, relationships, and legal precedents. This fundamental architectural difference means the system can reason about patent strategy, not just generate words.

There is still plenty of human judgement involved. The risk with any smart tool is that tired humans start accepting suggestions uncritically because they "sound right".

For me, the most promising use case is not a one-click patent generator, but a kind of second brain for teams dealing with portfolios. A practical guide to measuring LLM code generation quality

Why You Need an Eval Harness

Before spending a weekend building this, understand what problems it actually solves.

The Problem with Ad-Hoc Testing

Teams often test coding agents by manually reviewing generated samples, running one-off benchmarks, or collecting anecdotal feedback. This breaks down because:

  • You can't manually review thousands of generations
  • Different reviewers use different criteria (no reproducibility)
  • Changes that improve Python code might break JavaScript (no regression detection)
  • Teams cherry-pick examples that confirm their expectations (bias)

A proper evaluation framework gives you automated, repeatable benchmarks; multi-dimensional scoring; regression detection; confidence intervals; and fine-grained insights into where your agent struggles.

Architecture Overview

Four core components work together:

┌─────────────────┐ │ Task Registry │ ← Define what to test └────────┬────────┘ │ ▼ ┌─────────────────┐ │ Code Generator │ ← Your agent under test └────────┬────────┘ │ ▼ ┌─────────────────┐ │ Evaluators │ ← Multiple scoring dimensions └────────┬────────┘ │ ▼ ┌─────────────────┐ │ Aggregator │ ← Statistical analysis & reporting └─────────────────┘

Let's build each component.

Implementation Guide

Step 1: Define Your Tasks

Each task needs a prompt, test cases, and metadata. Here's the structure:

python 
from dataclasses import dataclass
from typing import List, Dict, Any

@dataclass
class TestCase:
    input: Any
    expected_output: Any
    
@dataclass
class Task:
    id: str
    prompt: str
    test_cases: List[TestCase]
    language: str  # 'python', 'javascript', etc.
    difficulty: str  # 'easy', 'medium', 'hard'
    tags: List[str]  # ['algorithms', 'sorting', ...]

Pro tip: Start with 20-30 well-chosen tasks covering common patterns. You can expand later. Quality beats quantity here.

Step 2: Build the Generator Interface

Your agent needs a consistent interface regardless of which LLM you're using underneath:

python 
class CodeGenerator:
    def generate(self, prompt: str, language: str) -> str:
        """
        Generate code from natural language prompt.
        Returns raw code without markdown formatting.
        """
        pass
        
# Concrete implementation
class ClaudeGenerator(CodeGenerator):
    def generate(self, prompt: str, language: str) -> str:
        response = client.messages.create(
            model="claude-sonnet-4-20250514",
            messages=[{
                "role": "user",
                "content": f"{prompt}\n\nLanguage: {language}"
            }]
        )
        return self._extract_code(response.content)

Step 3: Implement Evaluators

Each evaluator scores a different dimension. Here's a correctness evaluator:

python 
def evaluate_correctness(code: str, task: Task) -> float:
    """Returns score 0.0 to 1.0 based on test case pass rate"""
    passed = 0
    total = len(task.test_cases)
    
    for test in task.test_cases:
        try:
            result = execute_code(code, test.input)
            if result == test.expected_output:
                passed += 1
        except Exception:
            continue  # Failed test
    
    return passed / total

Security note: Execute generated code in sandboxed environments. Use Docker containers or dedicated VMs. Never run untrusted code directly on your machine.

Common Pitfalls

1. Test Case Leakage

Don't include exact test cases in your prompts. The agent should generalise from the problem description, not memorise specific inputs. Keep test cases completely separate from the task prompt.

2. Ignoring Statistical Significance

LLMs have variance. Run each task 3-5 times and report confidence intervals. A change from 75% to 77% accuracy might just be noise.

3. Over-Optimising for Your Eval Set

Teams start tuning prompts specifically to pass their eval suite. This defeats the purpose. Regularly rotate in new tasks and retire old ones.

4. Execution Timeouts

Set reasonable timeouts (5-10 seconds per test case). Some generated code will infinite loop. Don't let one bad generation hang your entire eval run.

From the inside, as a working actor and voiceover, it hits a more complicated set of nerves. Part of me thinks "great, I can licence this and earn money while I'm in rehearsal". There's proper identity verification for marketplace voices, contracts around usage, and a growing understanding that consent is not a box you tick once. But the social norms here are still forming. I don't think every performer needs to rush out and get cloned. But it's worth understanding what tools like ElevenLabs can and can't do, so that when a client mentions them you can respond from a position of knowledge rather than fear.

Beyond Steps: What The New Wave Of Wearables Is Really Measuring

Surveying Whoop, Oura and the Apple Watch, and asking what these metrics actually change in people's lives

[Wearables comparison visual]

The first generation of wearables sold us on steps. Ten thousand a day became a kind of secular commandment, printed on retail displays and baked into default goals. It was a simple story and, for a while, a useful one. Moving more is better than not moving. But as the hardware has matured, the marketing has shifted. The interesting devices now talk less about steps and more about strain, readiness and recovery.

Look at a typical day wearing an Apple Watch, a Whoop band and an Oura-style ring. The watch is excellent at nudging you through the day: close your rings, stand up now, take this call, pay for that coffee. It treats activity as something to sprinkle between tasks. Whoop and the ring, by contrast, are quietly watching the whole 24 hours. They care about heart rate variability, sleep stages, micro-movements. Apple Watch

The Daily Nudge

Treats activity as moments to capture between everything else. Close your rings. Stand now. Take this call. Excellent at keeping you minimally active throughout your waking hours.

Whoop

The Athlete's Oracle

Measures strain and recovery. Tells you whether to push hard or back off based on how stressed your system is. Popular with serious trainers who think in cycles.

Oura Ring

The Sleep Detective

Quietly watches your entire 24 hours with emphasis on rest. Heart rate variability, sleep stages, readiness scores. Asks whether you're recovered enough to perform.

For people who train seriously (or who simply like thinking in those terms) that's a useful reframing. It encourages you to see a brutal workout after a sleepless night as a bad trade, not a badge of honour. Recovery scores and strain targets are imperfect, but they nudge you towards a view of health that includes rest and mood, not just raw output. On days where my own readiness scores are low across devices, I can usually recognise the feeling without looking. The number just gives me permission to act on it.

You start asking your ring whether you're tired instead of listening to your own head

The risk is turning your body into a dashboard. With three devices strapped on, it's easy to outsource your intuition to metrics that are, at best, decent proxies. You notice patterns over weeks instead of agonising over individual days. "Every time I have late caffeine my sleep efficiency tanks." "My resting heart rate is higher for days after long-haul travel." You make small experiments and see what sticks.

The most valuable insights from wearables aren't the daily numbers. They're the patterns you notice over weeks and months. Late caffeine consistently wrecks your sleep. Long-haul travel takes three days to recover from, not one. That second drink at dinner costs you an hour of deep sleep.

These aren't revolutionary discoveries. Most people could guess them. But seeing the data makes it harder to ignore the connection between choices and consequences.

The bigger question for all these companies is whether they can keep moving beyond novelty. Counting steps was fun until everyone realised it didn't automatically make them healthier. Readiness scores and recovery bands will go the same way if they don't translate into changes people can actually feel in their lives.

A device has earned its place on your wrist or finger when you stop noticing it and quietly live a little better because it's there. The metrics matter less than whether you're sleeping more soundly, recovering more deliberately, or simply paying attention to signals your body was sending all along.

Ten thousand steps was never the point. The point was noticing when you weren't moving at all.

Equal Pay Law: What Supermarket Workers Need to Know

Last updated:

If you work in a supermarket and suspect you're being paid less than colleagues in the warehouse, you're not alone. Thousands of retail workers across the UK are joining group litigation claims under equal pay law, challenging pay disparities that have persisted for decades. Here's what you need to know.

What actually is equal pay law?

The principle is straightforward: workers doing equal work must receive equal pay, regardless of gender. The law has been around since the Equal Pay Act 1970, now part of the Equality Act 2010.

Here's the important bit: 'equal work' doesn't mean identical jobs. The law recognises three scenarios:

  • Like work: Jobs that are the same or broadly similar
  • Work rated as equivalent: Different jobs given equal value under a job evaluation study
  • Work of equal value: Different jobs requiring comparable skill, effort, and decision-making

That last category is where supermarket claims come in. Many retail workers, predominantly women, argue that their shopfloor roles require equivalent skill and responsibility to warehouse roles, predominantly done by men. Yet warehouse workers often earn £3 to £5 more per hour.

£6,000–£10,000

Potential annual lost earnings for shopfloor workers compared to warehouse staff at the same supermarket

Why are supermarkets being sued?

Four main reasons:

1. The pay gap is significant. We're not talking about a few pence. In many chains, warehouse and distribution staff earn between £3 and £5 more per hour than shopfloor workers. Over a year, that's £6,000 to £10,000 in lost earnings.

2. Separate pay structures. Supermarkets typically operate retail and warehouse work under different pay structures, arguing these are separate labour markets. Courts have increasingly rejected this when the work is found to be of equal value.

3. Gender imbalance. Retail shopfloors are about 70-80% women. Warehouse roles are predominantly men. This demographic split strengthens equal pay claims under UK law, which specifically protects against gender-based pay discrimination.

4. Precedents have been set. In 2021, a landmark ruling found in favour of shopfloor workers at a major UK supermarket, establishing that their work was indeed of equal value to warehouse roles. That opened the floodgates.

Am I eligible to join a claim?

You may be eligible if:

  • You work or worked for a supermarket in a customer-facing role (cashier, shelf stacker, customer service, bakery, deli, etc.)
  • There's a pay difference between your role and warehouse/distribution centre roles within the same organisation
  • You've worked there within the past six years (typical lookback period in England and Wales)
  • You believe your role requires comparable skill and responsibility to higher-paid colleagues

Group litigation allows thousands of workers to bring claims together, sharing legal costs and evidence. This makes it practically and financially viable to challenge large employers.

What does 'equal value' actually mean in practice?

This is the crux of most supermarket cases. Employment tribunals assess equal value by examining:

Skill requirements: Physical demands (lifting, standing, manual dexterity), customer service and communication skills, technical knowledge (tills, stock systems, product knowledge), problem-solving in real-time situations.

Responsibility: Handling cash and preventing loss, health and safety compliance, customer satisfaction and complaints, stock accuracy and merchandising standards.

Effort: Physical exertion (replenishing shelves, cleaning, till work), mental effort (multitasking, dealing with difficult customers), working conditions (customer-facing pressure vs warehouse environment).

What's critical: jobs don't need to be identical to be of equal value. A checkout operator juggling customer queries, scanning accuracy, cash handling, and promotional knowledge may be performing work of equal value to a forklift driver managing pallets in a warehouse. The tasks look different, but the value to the employer might be comparable.

How do group litigation claims actually work?

  1. Assessment and Registration

    You register your interest with a law firm specialising in employment law. They'll assess whether your circumstances meet the criteria for an equal pay claim.

  2. Evidence Gathering

    Your legal team gathers evidence about pay structures, job responsibilities, and any job evaluation studies the employer has conducted. This often involves expert witnesses who assess the value of different roles.

  3. Tribunal Proceedings

    Equal pay claims go through the employment tribunal system. In group litigation, 'lead cases' are selected to establish key legal principles. Once these cases succeed, they set precedents for the remaining claimants.

  4. Settlement or Judgement

    Many cases settle before reaching a full hearing, once employers recognise the strength of the claim. If the case proceeds to judgement and succeeds, the tribunal will order back pay and equalisation of future pay.

  5. Payment

    If successful, claimants receive back pay for up to six years of wage disparity, minus legal fees (usually on a no-win, no-fee basis).

What could I actually receive?

It depends on how long you've worked there and how large the pay gap is. Here's a rough example:

Say you've worked as a shelf stacker for 5 years, earning £12 per hour, while warehouse workers doing comparable work earned £15 per hour. That's a £3 per hour difference.

Working 37.5 hours per week for 52 weeks equals 1,950 hours per year. Over 5 years, that's 9,750 hours. At £3 per hour difference, you'd be owed £29,250 in back pay.

Obviously, every case is different. Some people are owed more, some less. But the figures can be substantial, especially if you've been with the same employer for several years.

Not because your product wasn't good enough. Not because the competitor undercut you. But because when your AE went on holiday, nobody else on the team knew where the conversation stood.

Here's the uncomfortable truth: your HubSpot hygiene is probably terrible. And it's costing you deals.

The Inbox Context Problem

Let's say Sarah on your sales team has been nurturing a prospect for six weeks. Emails back and forth about pricing, use cases, integration questions. It's all in her Gmail.

Then Sarah gets ill. Or goes on annual leave. Or worse, leaves the company.

The prospect emails back: "We're ready to move forward, but we need clarity on that API limitation we discussed."

Your team stares at HubSpot. The deal is there. The company is there. But the context? Nowhere. Someone has to piece together the conversation from Sarah's email archives (if they have access) or awkwardly ask the prospect to repeat themselves.

You look unprepared. You lose credibility. Sometimes, you lose the deal.

This happens because most teams treat HubSpot as a logging tool, not a source of truth. They update it when they remember. They copy-paste selectively. And critical context stays trapped in individual inboxes.

Why Manual Logging Fails

"Just log your emails" is not a system. It's a hope.

Here's what actually happens:

1. Your rep finishes a call. They're pumped. They update HubSpot with notes.
2. The prospect emails a follow-up question. Your rep responds quickly (good!) but forgets to log it (bad).
3. Two days later, the prospect replies with more questions. Your rep answers. Still doesn't log.
4. A week passes. Your rep vaguely remembers they should log something, but can't recall what. They skip it.
5. The deal stalls. Your manager asks, "What's the status?" Your rep checks HubSpot. It says "Proposal Sent" from three weeks ago.

The context gap is now a chasm.

This isn't because your reps are lazy. It's because manual logging is friction. Every second spent copying emails into HubSpot is a second not spent selling. So it gets deprioritised. Always.

The Auto-Log Solution

This is where auto-logging transforms everything.

Instead of asking reps to manually copy emails into HubSpot, integrate your email system directly. Every email to or from a contact associated with a HubSpot record gets logged automatically. No human intervention. No gaps.

Suddenly:

→ Sarah's six weeks of email context is in HubSpot, not just her inbox
→ When she's out, anyone on the team can pick up the thread
→ Your manager can actually see deal progression in real-time
→ Handoffs between SDRs and AEs are seamless (all context is already there)
→ Historical deals become learning resources (new reps can see how deals were won)

The Compound Effect

Auto-logging doesn't just fix one deal. It fixes your entire sales operation.

1. Handoffs become frictionless

When an SDR qualifies a lead and hands off to an AE, the AE sees every email. No "can you catch me up?" calls. Just read the history and continue.

2. Deal reviews become productive

Your Monday deal review used to be reps reading from memory. Now your manager opens HubSpot, sees the actual email thread, and asks specific questions:

"I see the prospect mentioned budget concerns on Thursday. How did you address that?"

Accountability skyrockets.

3. Onboarding accelerates

New reps can study closed deals. They see the exact email sequences that worked. They learn objection handling from real conversations, not training decks.

4. Reporting becomes accurate

Your activity reports finally reflect reality. You're not guessing how many touchpoints a deal took. You know.

5. Customer success has context

When a deal closes and moves to CS, the entire email history comes with it. Your CS team doesn't start from scratch. They know what was promised, what concerns were raised, what the customer cares about.

How to Actually Implement This

If you're on HubSpot (and Gmail or Outlook), this is embarrassingly easy:

Step 1: Enable email integration

HubSpot's native email integration is free. Go to Settings > Integrations > Email, connect your Gmail or Outlook. Done.

Step 2: Set logging preferences

Choose what gets logged automatically:
• All emails to/from HubSpot contacts (recommended)
• Only emails you BCC to HubSpot (manual, defeats the purpose)
• Emails from specific domains only (useful for filtering out internal noise)

I recommend "all emails to/from contacts." Yes, it's aggressive. Yes, it logs everything. That's the point.

Step 3: Audit and clean

Once auto-logging is live, you'll see the gaps in your current data. Fix them:
• Ensure every prospect has a HubSpot contact record
• Associate contacts with the right companies and deals
• Train reps to create records before emailing (or use automation to do it)

Step 4: Enforce at team level

Make it non-negotiable. If someone sends a deal-related email outside HubSpot and it doesn't log, that's a problem to fix immediately.

Your CRM should be the single source of truth for customer relationships.

Not someone's inbox.
Not a Google Doc.

→ Your best practices are documented and repeatable

This doesn't happen by accident. It happens when you remove friction from logging.

Auto-logging emails is the lowest-hanging fruit. Start there.

Then move to:
• Auto-logging calls (HubSpot calling + transcription)
• Auto-logging meetings (Zoom/Teams integration)
• Auto-creating tasks from emails (AI-powered, based on intent)
• Auto-updating deal stages (based on email content analysis)

All of this is possible today. Most teams just haven't implemented it.

Amazon Q Developer: The AI Coding Assistant That Actually Understands Your Codebase

Published:

The AI coding assistant market is crowded. GitHub Copilot, Cursor, Codeium, and now Amazon Q Developer all promise to accelerate development with AI-powered code suggestions. But in a landscape where every tool claims to 10x productivity, what makes Amazon Q Developer worth evaluating?

After testing Q Developer across multiple projects, teams, and codebases, we've found it excels in one critical area where competitors struggle: understanding and working within the specific context of your existing codebase, AWS infrastructure, and organisational coding standards.

Here's what you need to know.

What Is Amazon Q Developer?

Amazon Q Developer is AWS's AI-powered coding assistant, designed to help developers write, review, and refactor code faster. It's available as IDE extensions for VS Code, JetBrains IDEs, and AWS Cloud9, plus direct integration with the AWS Management Console.

Unlike generic AI coding tools that work in isolation, Q Developer is tightly integrated with AWS services. It understands:

  • Your existing AWS infrastructure (Lambda functions, ECS services, RDS databases)
  • AWS SDK patterns and best practices
  • Your organisational code repositories (when configured)
  • Security and compliance requirements specific to AWS environments

This contextual awareness is Q Developer's differentiator. It's not just autocomplete on steroids; it's a tool that understands where your code lives and how it connects to your infrastructure.

Key Features That Matter

1. Inline Code Suggestions (with Real Context)

Like Copilot, Q Developer provides real-time code completions as you type. But here's where it differs: when you're working in an AWS Lambda function, Q Developer knows you're in Lambda. It suggests code that:

  • Uses the correct Lambda handler signature
  • References environment variables you've actually defined in your Lambda config
  • Imports libraries from layers you've attached
  • Follows AWS SDK patterns specific to your Node.js, Python, or Java runtime

This isn't hypothetical. When you type `const s3 = new`, Q Developer doesn't just suggest `AWS.S3()`; it suggests the correct initialisation pattern for your specific SDK version, region configuration, and IAM permissions.

2. Chat Interface for Complex Questions

The Q Developer chat panel (accessible via `/q` in your IDE) lets you ask questions about your codebase and AWS setup. Examples from real use:

  • "How do I configure this Lambda to access the RDS instance in my VPC?"
  • "What's the best way to batch process these S3 files without hitting rate limits?"
  • "Why is my DynamoDB scan running slowly, and how can I refactor it?"

Q Developer doesn't just provide generic answers. It references your actual infrastructure, suggests specific changes, and can even generate code modifications for you to review and apply.

3. Codebase Indexing (Beta)

This is the feature that changes everything. When you connect Q Developer to your Git repository (GitHub, GitLab, or Bitbucket), it indexes your codebase. This means:

  • Code suggestions reference your internal libraries and utilities
  • It understands your team's naming conventions and patterns
  • It suggests refactors that align with your existing architecture
  • It can answer questions like "Where do we handle authentication in this service?"

Testing this with a 500,000-line monorepo, Q Developer accurately identified authentication middleware used across 30+ Express routes and suggested consistent implementations for a new endpoint.

4. Security Scanning and Remediation

Q Developer includes built-in security scanning that identifies vulnerabilities as you code. It catches:

  • Hardcoded secrets and API keys
  • SQL injection vulnerabilities
  • Cross-site scripting (XSS) risks
  • Insecure dependencies with known CVEs
  • AWS-specific misconfigurations (overly permissive IAM policies, public S3 buckets)

What's valuable: it doesn't just flag issues. It suggests fixes, often with one-click remediation. For example, detecting a hardcoded API key, it suggests moving it to AWS Secrets Manager and generates the code to retrieve it securely.

5. Testing and Documentation Generation

Q Developer can generate:

  • Unit tests for functions (with mocking setup for AWS services)
  • Documentation comments in your preferred format (JSDoc, Python docstrings, etc.)
  • README files for repositories or microservices

The test generation is particularly strong. For a Lambda function that reads from SQS, processes data, and writes to DynamoDB, Q Developer generated:

  • Mock SQS event payloads
  • DynamoDB client mocking setup using `aws-sdk-mock`
  • Edge case tests (empty queues, malformed messages, DynamoDB failures)

Were the tests perfect? No. Did they provide a solid scaffold that saved 30 minutes of boilerplate writing? Absolutely.

Where Q Developer Excels

AWS-Native Development

If your stack is AWS-heavy, Q Developer is unmatched. It understands:

  • **AWS CDK and CloudFormation**: Suggests infrastructure-as-code patterns, validates resource configurations, and catches common misconfigurations before deployment.
  • **Serverless patterns**: Knows how to structure Lambda functions, API Gateway integrations, and Step Functions workflows.
  • **Cost optimisation**: Suggests more efficient service choices (e.g., "Use S3 Select instead of downloading entire objects").

Example: while building a data pipeline, Q Developer suggested switching from a Lambda + S3 polling architecture to EventBridge + Lambda triggers, reducing latency and eliminating unnecessary S3 API calls. This wasn't just code completion; it was architectural guidance.

Legacy Code Modernisation

Q Developer is excellent for understanding and refactoring older codebases. When working with a legacy Node.js service using callbacks, Q Developer:

  • Identified callback hell patterns
  • Suggested Promise-based refactors
  • Converted functions to async/await with proper error handling
  • Flagged deprecated AWS SDK v2 calls and suggested v3 equivalents

This is invaluable for teams maintaining brownfield projects.

Team Consistency

With codebase indexing, Q Developer learns your team's patterns. If your team consistently uses a specific error handling pattern or a particular logging library, Q Developer will suggest code that matches these conventions.

This reduces code review friction. When every developer's AI assistant is trained on the same codebase, PR diffs become more uniform, and onboarding new developers is faster.

Where Q Developer Falls Short

No tool is perfect. Here's where Q Developer struggles:

1. Non-AWS Workloads

If you're primarily working with GCP, Azure, or on-premises infrastructure, Q Developer offers less value. It's AWS-aware but not multi-cloud fluent. For polyglot cloud environments, GitHub Copilot or Codeium may be better fits.

2. Frontend and Mobile Development

Q Developer's strengths are backend and infrastructure code. For React, Vue, Swift, or Kotlin development, it works but doesn't shine. It lacks the frontend-specific context and component library knowledge that tools like Cursor provide.

3. Latency

Code suggestions occasionally lag, particularly in large files or when codebase indexing is enabled. This isn't unusual for AI coding tools, but it's noticeable compared to Copilot's snappier performance.

4. Learning Curve for Chat Features

The chat interface is powerful but requires knowing how to ask good questions. Developers used to Stack Overflow-style queries may need to adjust their approach to get valuable answers.

Pricing: Is It Worth It?

Amazon Q Developer offers two tiers:

  • **Free tier**: Inline code suggestions, limited chat interactions (50 per month), basic security scans
  • **Pro tier** (£15/user/month): Unlimited chat, codebase indexing, advanced security scanning, customisation features

For comparison:

  • **GitHub Copilot**: £8/month (individual), £15/user/month (business)
  • **Cursor**: £16/month (includes premium models)
  • **Codeium**: Free for individuals, enterprise pricing on request

Q Developer's Pro tier is competitively priced, particularly for teams already invested in AWS. The codebase indexing and AWS integration justify the cost if your development is infrastructure-heavy.

Who Should Use Amazon Q Developer?

  • Teams building on AWS (Lambda, ECS, RDS, DynamoDB, etc.)
  • DevOps engineers managing infrastructure-as-code (CDK, CloudFormation, Terraform)
  • Organisations with large, existing codebases that need contextual assistance
  • Security-conscious teams requiring built-in vulnerability scanning
  • Pure frontend or mobile development teams
  • Multi-cloud or cloud-agnostic organisations
  • Individual developers looking for the cheapest option (GitHub Copilot's free tier is more generous)

Getting Started: A Practical Approach

If you're evaluating Q Developer, here's a pragmatic rollout plan:

Week 1: Install and Observe

  • Install the IDE extension
  • Use the free tier to get a feel for inline suggestions
  • Don't configure anything yet; just observe how often suggestions are useful

Week 2: Connect Your Codebase

  • Enable codebase indexing for one repository (choose a mature, well-structured one)
  • Ask Q Developer questions about existing code patterns
  • Evaluate whether suggestions improve in relevance

Week 3: Security and Testing

  • Run security scans on your codebase
  • Use Q Developer to generate tests for untested functions
  • Assess the quality of generated tests (do they pass? do they cover edge cases?)

Week 4: Team Pilot

  • Upgrade 5–10 developers to Pro tier
  • Collect feedback on productivity impact
  • Measure code review time and bug rates

If after four weeks you're seeing measurable improvements in developer velocity, reduced security vulnerabilities, or faster onboarding, Q Developer is likely worth adopting more broadly.

The Bottom Line

Amazon Q Developer isn't a revolutionary tool that will replace developers. It's an evolutionary one that makes existing workflows faster, particularly for teams deeply embedded in AWS.

Its strength is contextual awareness. When your AI assistant understands your infrastructure, your codebase, and your team's conventions, suggestions become useful rather than generic. That's the promise of Q Developer, and in practice, it largely delivers.

Is it better than GitHub Copilot? For AWS-native development, yes. For everything else, it depends. Try the free tier, evaluate it against your actual work, and decide based on what you're building, not on marketing promises.

AI coding assistants are tools, not magic. The question isn't whether Q Developer is revolutionary; it's whether it makes your team's specific work faster and better. For many AWS-focused teams, the answer is yes.
